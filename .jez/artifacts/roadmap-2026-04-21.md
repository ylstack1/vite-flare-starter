## Ordering rationale

1. **Commit first** ‚Äî prevents merge pain if laptop breaks or Jez swaps machines
2. **Audit fixes before new features** ‚Äî H1/H2/M1 are user-visible polish blockers
3. **Map polish** is cheap and closes the current feature arc
4. **Email before RAG** ‚Äî email unblocks auth flows for any fork that enables email/password. RAG is a more advanced feature that smaller forks may not need.
5. **RAG before Connectors** ‚Äî the RAG pipeline becomes the ingestion target for connector-sourced documents (Phase 5.6)
6. **Connectors last** ‚Äî largest piece, needs the other phases to be stable

## Principles carried through this roadmap

- **Don't break the starter's fork-friendly story** ‚Äî every feature must work with minimum config and degrade gracefully when bindings/secrets are missing
- **Every phase produces a demo-ready artefact** ‚Äî no half-done phases
- **Document-as-you-go** ‚Äî update CLAUDE.md, `.dev.vars.example`, and inline comments as each phase lands
- **Security-conscious defaults** ‚Äî destructive tools `ask` by default, no tokens in logs, encrypted storage for OAuth tokens
- **Pattern-library discipline** ‚Äî whatever ships teaches the forker how to build similar capabilities, not just "use this feature"
